# 自动暂停设计（5 分钟无操作）

## 背景

当前应用在计时模式下会持续倒计时，用户希望当电脑超过 5 分钟没有任何操作时，自动暂停计时，避免离开工位时仍继续累计倒计时。

本次设计只覆盖“自动暂停”能力，不包含自动恢复、额外弹窗提示或跨平台支持。

## 目标

- 当应用处于计时中时，若系统级无操作时间达到 5 分钟，自动暂停当前倒计时。
- 自动暂停后，倒计时停留在当时剩余时间，等待用户手动恢复。
- 自动暂停需要写入运行记录，并在运行记录窗口中显示为中文原因“自动暂停”。

## 非目标

- 不实现自动恢复。
- 不增加新的设置项或可配置阈值。
- 不增加额外提示框、托盘气泡或声音提醒。
- 不实现 macOS / Linux 支持，本次只覆盖 Windows。

## 方案对比

### 方案 A：在现有轮询中检查系统空闲时间（推荐）

在 `AppController._check_due()` 的现有 1 秒轮询中，增加一次系统空闲时间检查。当：

- `scheduler.is_running == True`
- 当前没有提醒弹窗显示
- 系统空闲秒数 `>= 300`

则直接走现有暂停链路。

**优点：**
- 与现有架构最贴合，改动小。
- 复用现有暂停行为与 UI 刷新逻辑。
- 容易在测试中替换系统空闲时间来源。

**缺点：**
- 空闲时间读取逻辑会进入控制器，需要额外抽出一个小接口以保持可测性。

### 方案 B：新增独立空闲监控器对象

新增一个专门负责读取系统空闲时间的对象，由它发出“应自动暂停”信号，`AppController` 订阅后再暂停。

**优点：**
- 职责更清晰。
- 后续扩展自动恢复、空闲告警时更自然。

**缺点：**
- 对当前单一需求来说偏重。

### 方案 C：只根据应用自身事件近似判断

不读取系统 API，只统计应用窗口自己的输入事件。

**优点：**
- 实现最简单。

**缺点：**
- 与需求不符。用户在别的软件里操作时会被误判为“无操作”。

## 最终方案

采用方案 A。

### 设计要点

1. 在 `AppController` 中增加一个系统空闲秒数入口，例如 `_system_idle_seconds()`。
2. 在 Windows 上通过系统 API 读取“最后输入时间”并计算空闲秒数；实现时优先使用 `GetTickCount64` 这类不会因系统长时间运行而产生 32 位回绕问题的计时来源。
3. 在 `_check_due()` 中，自动暂停判断的插入点应位于“提醒弹窗已显示”的 guard 之后、正常 due 判断之前，避免同一个 tick 内既弹提醒又自动暂停。
4. 自动暂停判定应明确依赖当前调度器状态不变量：`is_running == True` 且 `is_paused == False`。当前 `ReminderScheduler` 中这两个状态互斥，但 spec 明确写出该约束，避免后续改动引入重复触发。
5. 满足条件时，调用一个专门的自动暂停入口 `_auto_pause_countdown()`，而不是直接复用普通按钮暂停的结束原因。
6. `_auto_pause_countdown()` 的动作顺序与现有 `pause_countdown()` 保持一致：先收口当前运行 segment，结束原因为 `auto_pause`；再调用调度器暂停；最后刷新主页面状态。
7. 自动暂停后：
   - 调用调度器暂停
   - 冻结剩余时间
   - 刷新主页面状态
   - 写入一条运行记录，`end_reason = "auto_pause"`
8. 运行记录窗口中把 `auto_pause` 翻译为 `自动暂停`。
9. 手动恢复与自动暂停的关系先保持最小语义：只有当用户产生新的真实输入后，系统空闲时间才会归零；因此如果用户在仍被系统判定为空闲的情况下立即恢复计时，下一次轮询可能再次自动暂停。本次不为该情况额外增加 debounce 机制。
10. 如果自动暂停触发时 `_active_run_started_at is None`，则不写运行记录，但仍允许调度器进入 paused 状态；这是为了与当前 `_finish_active_run()` 的静默保护保持一致。
11. 本次不顺带扩展退出程序时的运行记录收口，`quit_app()` 的结束原因记录仍属于后续功能。

### 设计边界补充

- `history_dialog.py` 中的 `RunHistoryDialog` 通过委托 `RunHistoryStore.translate_end_reason()` 完成文案翻译，因此实现时只需要在 `run_history.py` 的翻译表中新增 `auto_pause` 即可。
- `ReminderScheduler` 本身不需要改动；自动暂停只发生在控制器层，对调度器仍复用现有 `pause(now_ts)` 语义。

## 代码结构调整

### `app_controller.py`

新增：
- `_system_idle_seconds()`：返回当前系统空闲秒数
- `_should_auto_pause(idle_seconds, threshold_seconds=300)`：封装自动暂停判定
- `_auto_pause_countdown()`：执行自动暂停动作

修改：
- `_check_due()`：在现有 due 检查前增加自动暂停判断
- `pause_countdown()` 或底层记录收口函数：支持记录 `auto_pause`

### `run_history.py`

修改：
- `translate_end_reason()` 增加：
  - `"auto_pause": "自动暂停"`

### 测试

重点覆盖：
- 运行中且空闲超 5 分钟时自动暂停
- 未运行时不自动暂停
- 已暂停状态不重复触发
- 提醒弹窗显示时不自动暂停，并确认 `_check_due()` 中的 guard 顺序正确
- 自动暂停写入 `auto_pause` 记录
- 运行记录窗口把 `auto_pause` 显示为 `自动暂停`
- `_system_idle_seconds()` 在系统 API 读取失败时返回 `0`
- 自动暂停触发时 `_active_run_started_at is None` 不应崩溃
- `RunHistoryStore.translate_end_reason("auto_pause")` 的独立单测
- 人工恢复后，如果系统仍判定为空闲，下一次轮询再次自动暂停属于本次接受行为，并需有测试体现该语义

测试注入方式优先使用：
- `controller._system_idle_seconds = MagicMock(return_value=300)`
- `controller._now_ts = MagicMock(...)`
这样可避免直接 patch Windows API，提高测试稳定性。

## 数据流

1. `QTimer` 每秒触发 `AppController._check_due()`
2. `_check_due()` 先执行现有“提醒弹窗已显示”保护；若弹窗已显示，则直接返回
3. `_check_due()` 调用 `_system_idle_seconds()`
4. 若达到 300 秒且当前正在运行，并且没有提醒弹窗，则进入 `_auto_pause_countdown()`
5. `_auto_pause_countdown()` 记录 segment，结束原因为 `auto_pause`
6. 调度器进入 paused 状态，剩余时间被冻结
7. 主界面显示 `已暂停`
8. 运行记录窗口刷新后显示新的自动暂停记录
9. 若不满足自动暂停条件，再继续执行原有 due 判断逻辑

## 边界行为

- 如果机器长时间开机，空闲检测仍应正确工作，不应因 tick 计数回绕导致误判。
- 如果用户恢复计时但系统空闲状态尚未被新的真实输入清零，下一次轮询再次自动暂停是本次接受行为。
- 如果 `_active_run_started_at` 缺失，则自动暂停只改变调度器状态，不补写伪造的运行记录。
- 如果应用启动后系统已经空闲超过 5 分钟，则首个满足条件的轮询可能直接触发自动暂停；该行为属于当前规则自然结果。
- 如果系统空闲时间读取失败，则视为 `0`，不触发自动暂停。
- 退出程序时不为当前运行段补写 `quit` 记录，本次不扩展该语义。

## 错误处理

- 如果系统空闲时间读取失败，则返回 `0`，等价于“不触发自动暂停”。
- 该失败不应中断现有倒计时流程。
- 不新增用户可见错误提示，避免把系统 API 细节暴露到 UI。
- 系统空闲计时实现应避免 32 位 tick 回绕带来的误判。

## 平台范围

- 本次只实现 Windows。
- Windows 之外的平台保持现状，不触发自动暂停。

## 验证计划

1. 单测覆盖自动暂停判定与记录行为。
2. 单测覆盖运行记录翻译文案。
3. 单测通过 `controller._system_idle_seconds = MagicMock(return_value=300)` 注入空闲时间。
4. 运行应用后手动验证：
   - 启动计时
   - 模拟或注入空闲时间达到 300 秒
   - 确认自动进入暂停状态
   - 确认运行记录窗口可见“自动暂停”
   - 确认提醒弹窗场景不会和自动暂停在同一个 tick 冲突
